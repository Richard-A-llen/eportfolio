JSON injection attack

Initial post

As others have already noted in this discussion injection attacks are one of the top threats as identified by the OWASP. According to OWASP 2021 report on the most critical web application security risks the injection attack ranked 3rd and 94% of applications had some type of vulnerability to an injection attack (Imperva, 2021). I first encountered the NoSQL injection attack during last module NoSQL alternative discussion, and I wanted to continue my research for this discussion. NoSQL injection attacks are a threat because they manipulate web based applications by injecting malicious code into an application to alter it's normal execution leading to loss of data, modification of data and denial of service. (Welekwe, 2020).

Web applications often use MongoDB and other NoSQL databases, often include input forms, an attacker can use weaknesses from areas such as input forms to embed malicious code into a NoSQL queries.

The injection may take the form of a JSON file. JSON files are used by web applications to communicate with databases to retrieve information – they can also be a vulnerable point for an attack. If an attacker can hijack a JSON file and exploit a vulnerability they may bypass firewalls and endpoint defences so the attacker can manipulate the database as an authorised user (Eassa et al, 2018)

Much of my flowchart is based on the demonstration of a JSON injection attack (Yang, 2021) by Ethical hacker and Cybersecurity consultant Loi Liang Yang please find link to the video in my references.

Hi Greg. This is an important threat to be aware of as any user of a web application could fall victim as you have mentioned there are multiple ways a user's session can become compromised. It is easy to fall victim to a sniffing attack when using unsecured Wi-Fi networks. Packet sniffing tools can steal data and sensitive information transmitted over the network of an unsecure network. Malicious actors can take advantage of already unsecure networks or create there own, often disguised as free public Wi-Fi networks (CISOMAG, 2021). 

Ways to prevent oneself falling victim to a sniffing session attack are to avoid free unsecure Wi-Fi networks and to encrypt all incoming and outgoing communication using a virtual private network (VPN). A VPN would make it difficult for the attacker to decrypt the packet's data, likely the attacker would just find what looked to them like random data of no value(CISOMAG, 2021).

Summary Post

Missing from my original model was an explanation of how the attacker gets access to the data from the web application. Attackers can scan websites for vulnerabilities using penetration testing tools such as Burp Suite. These are sophisticated tools which can intercept traffic between a client and a server. Effectively acting as a man in the middle between the browser and the website – allowing the user to intercept view and modify all traffic in both directions (portswigger.net, 2021).

In the case of the JSON injection attack I've attempted to model, although penetration testing tools are capable of capturing all the data flowing between a browser and a server because of the volume of data rendering all raw html and sifting through it data manually would be time consuming, the attacker would either select the type of data to intercept via the tool (if the option was available) or to manually search through all the data using a keyword. For example searching “JSON” in the raw response data coming from the server would allow an attack to locate potential JSON files for attack (Yang, 2021).

JSON files cause vulnerabilities because they send code which will be read by the server, and because JSON file are versatile it straight forward to put in an array or key value that the server isn't expect and which could cause an unexpected result (LeCheminant, 2015). As Rachel mentioned a lack of JSON-sanitizer would make a system far more vulnerable, I have included this in my new attack model. Wang and Greg also both suggested approaches which involve monitoring the incoming code for JSON injection attacks, which I agree with as a appropriate solution and have included in my diagram. A JSON sanitizer fixes incoming constructs which could result in unwanted responses from the server and turns them into valid JSON – as demonstrated by OWASP sanitizer available on Github(Samuel, 2021). Thanks to all those who responded to my diagram.
